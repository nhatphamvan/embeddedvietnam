CH / 01
BSW Architecture

Tổng quan MCU Driver trong BSW

MCU Driver là một trong những module MCAL nền tảng nhất trong AUTOSAR Classic Platform. Theo SWS, module này cung cấp các service cho khởi tạo vi điều khiển cơ bản, quản lý power-down, reset, và các chức năng đặc thù MCU mà các MCAL driver khác cần dùng.

Vị trí trong layered architecture

MCU Driver nằm ở tầng Microcontroller Abstraction Layer (MCAL), ngay sát hardware. Nó được truy cập trực tiếp bởi tầng Service Layer — đặc biệt là module EcuM (ECU State Manager) trong quá trình startup, và bởi BswM khi cần chuyển mode hoạt động.

┌─────────────────────────────────────────────────┐ │ Application Layer (SWC) │ ├─────────────────────────────────────────────────┤ │ RTE — Runtime Environment │ ├─────────────────────────────────────────────────┤ │ Service Layer: EcuM ◄──┐ │ │ BswM ◄──┤ calls Mcu_*() │ ├──────────────────────────│──────────────────────┤ │ ECU Abstraction Layer │ │ ├──────────────────────────│──────────────────────┤ │ MCAL: Mcu Driver ◄────────┘ │ │ Port · Dio · Gpt · Spi · Can · Adc │ ├─────────────────────────────────────────────────┤ │ Hardware: STM32F4 / STM32F1 │ └─────────────────────────────────────────────────┘

Chức năng chính

  • Initialization services — cấu hình clock tree, RAM sections, modes hoạt động của MCU.
  • Power-down support — chuyển MCU vào các low-power mode (Sleep / Stop / Standby).
  • Reset handling — đọc reset reason và thực hiện software reset.
  • PLL management — start PLL, phân phối PLL clock tới hệ thống bus, kiểm tra trạng thái lock.

Ranh giới với startup code

Theo SWS, một số việc không thuộc MCU Driver mà phải làm trong start-up code (rất MCU-specific):

  • Khởi tạo base address cho interrupt và trap vector table.
  • Khởi tạo user stack pointer.
  • Khởi tạo external memory (nếu có).
  • Đọc trạng thái cấp nguồn ECU/µC — đây là trách nhiệm của upper layer (EcuM/Hardware abstraction), không phải MCU Driver.
Theo SWS_Mcu R25-11: "The MCU driver provides services for basic microcontroller initialization, power down functionality, reset and microcontroller specific functions required by other MCAL software modules." MCU Driver không chịu trách nhiệm bật/tắt nguồn cho ECU hay vi điều khiển — đó là việc của upper layer.
Mô hình tư duy đơn giản: Nếu bạn quen với HAL/CMSIS trên STM32, MCU Driver trong AUTOSAR chính là "RCC + PWR + một phần SCB" được bọc lại theo chuẩn AUTOSAR API. Mọi thứ trong RCC (clock setup, PLL config) và PWR (low-power mode) đều phải route qua MCU Driver thay vì code thẳng vào register.

Mô hình clock setting

MCU Driver dùng khái niệm "clock setting" như một preset cấu hình clock hoàn chỉnh. Mỗi clock setting có ID duy nhất (McuClockSettingId) và chứa toàn bộ tham số PLL, prescaler, clock source... cần thiết để đưa MCU đến một cấu hình clock cụ thể.

Trong tool (EB tresos / DaVinci), bạn định nghĩa N clock setting (ví dụ: "100MHz_HSE", "16MHz_HSI_LowPower"...). Lúc runtime, gọi Mcu_InitClock(0) để áp setting 0, Mcu_InitClock(1) để áp setting 1. Điều này cho phép dynamic frequency scaling mà không cần đụng tới register.

Mô hình MCU mode

MCU mode là các trạng thái hoạt động khác nhau của vi điều khiển — chủ yếu liên quan đến low-power. Mode điển hình:

  • RUN — chế độ hoạt động bình thường, CPU và clock chính ON.
  • SLEEP — CPU dừng, peripheral clock có thể vẫn chạy (WFI/WFE).
  • STOP — hầu hết clock dừng, chỉ LSI/LSE còn chạy.
  • STANDBY — toàn bộ digital off trừ backup domain.

Mỗi mode có ID riêng định nghĩa trong container McuModeSettingConf và áp dụng qua Mcu_SetMode(modeId).

SWS_Mcu_00126: Hàm Mcu_Init luôn nhận pointer đến config struct, kể cả với VARIANT-PRE-COMPILE (khi không cần config set). Trong PC variant, pass NULL_PTR. Đây là quy tắc bất khả kháng — đừng thay đổi prototype của Mcu_Init.

Configuration variants

AUTOSAR định nghĩa 3 variant về thời điểm cấu hình được fix. MCU Driver hỗ trợ tất cả nhưng thực tế trên STM32F4/F1 thường dùng Pre-compile là chính.

VARIANT-PRE-COMPILE
PC
Config fix tại compile time. Generated code chứa hard-coded values. Mcu_Init nhận NULL_PTR. Phù hợp 90% project STM32.
VARIANT-LINK-TIME
LT
Config fix lúc link. Cho phép có nhiều cấu hình trong cùng object file. Ít dùng trên STM32.
VARIANT-POST-BUILD
PB
Config thay đổi sau khi flash (qua reflash partition). Dùng khi cùng firmware chạy trên nhiều variant ECU.

CH / 04
Service interface

API Service Interface

Theo SWS R25-11, MCU Driver expose 11 service chính. Mỗi service có Service ID cố định (dùng cho DET error reporting). Tên service ID hex thường dùng trong implementation:

SID 0x00 Mcu_Init
Synchronous Non Reentrant Standard
Prototype
void Mcu_Init(const Mcu_ConfigType* ConfigPtr);
Mô tả

Khởi tạo MCU Driver. Đây là service đầu tiên phải gọi trước mọi service khác. Trong VARIANT-PRE-COMPILE, pass NULL_PTR theo SWS_Mcu_00126.

Khi nào gọi

EcuM gọi rất sớm trong startup sequence, ngay sau khi start-up code hoàn tất (vector table, stack pointer, external memory init). Trên STM32, gọi ngay sau khi exit từ Reset_Handler và trước Port_Init.

Constraints
  • Không gọi lại nếu MCU đã được init (gây MCU_E_ALREADY_INITIALIZED nếu DET on).
  • Không gọi từ ISR.
SID 0x01 Mcu_InitRamSection
Synchronous Non Reentrant Standard
Prototype
Std_ReturnType Mcu_InitRamSection(Mcu_RamSectionType RamSection);
Mô tả

Fill một vùng RAM với giá trị mặc định, theo cấu hình trong McuRamSectorSettingConf. Theo SWS_Mcu_00011, ghi từ McuRamSectionBaseAddress đến + McuRamSectionSize - 1 với McuRamDefaultValue, mỗi lần ghi McuRamSectionWriteSize byte.

Khi nào gọi

Sau Mcu_Init (SWS_Mcu_00136). Dùng để clear RAM trước khi sử dụng — tránh side-effect từ memory bị nhiễu (ECC, soft error). Trên STM32, ít dùng vì RAM tích hợp không có ECC và start-up code thường đã zero-init .bss.

SID 0x02 Mcu_InitClock
Synchronous Non Reentrant Standard
Prototype
Std_ReturnType Mcu_InitClock(Mcu_ClockType ClockSetting);
Mô tả

Khởi tạo PLL và các clock option đặc thù của MCU theo ClockSetting (ID đã định nghĩa trong McuClockSettingConfig). Theo SWS_Mcu_00137, function khởi tạo PLL và các MCU-specific clock option; theo SWS_Mcu_00138, function bắt đầu quá trình PLL lock.

Constraints
  • Chỉ available khi McuInitClock = TRUE trong config (theo SWS_Mcu_00210).
  • Hàm này khởi động PLL nhưng không chờ lock — phải gọi Mcu_GetPllStatus để kiểm tra.
  • Trước khi switch SYSCLK sang PLL, phải gọi Mcu_DistributePllClock.
Trên STM32F4/F1

Implementation thường wrap quanh RCC: ghi PLLCFGR (F4) hoặc CFGR (F1), bật PLLON, set flash latency, set bus prescalers. Chi tiết khác biệt xem chương 11.

SID 0x03 Mcu_DistributePllClock
Synchronous Non Reentrant Standard
Prototype
Std_ReturnType Mcu_DistributePllClock(void);
Mô tả

Kích hoạt phân phối PLL clock lên hệ thống bus của MCU — tức là switch SYSCLK từ HSI/HSE sang PLLCLK. Phải gọi sau khi Mcu_GetPllStatus trả về MCU_PLL_LOCKED.

Constraints
  • Service này không available nếu McuNoPll = TRUE (MCU không có PLL hoặc PLL tự lock không cần SW).
SID 0x04 Mcu_GetPllStatus
Synchronous Reentrant Standard
Prototype
Mcu_PllStatusType Mcu_GetPllStatus(void);
Mô tả

Trả về trạng thái lock của PLL: MCU_PLL_LOCKED / MCU_PLL_UNLOCKED / MCU_PLL_STATUS_UNDEFINED. Upper layer (EcuM) thường polling hàm này trong vòng lặp với timeout.

Constraints
  • Reentrant — có thể gọi từ multiple context (main + ISR).
  • Nếu McuNoPll = TRUE, trả về MCU_PLL_STATUS_UNDEFINED.
SID 0x05 Mcu_GetResetReason
Synchronous Non Reentrant Standard
Prototype
Mcu_ResetType Mcu_GetResetReason(void);
Mô tả

Trả về lý do reset gần nhất của MCU: MCU_POWER_ON_RESET, MCU_WATCHDOG_RESET, MCU_SW_RESET, hoặc MCU_RESET_UNDEFINED. EcuM dùng giá trị này để quyết định post-reset behavior (ví dụ: vào safe-mode nếu watchdog reset).

Khi nào gọi

Gọi rất sớm trong startup sequence, sau Mcu_Init. Phải clear reset flag sau khi đọc — nếu không, lần reset sau sẽ thấy flag chồng chất.

SID 0x06 Mcu_GetResetRawValue
Synchronous Non Reentrant Standard
Prototype
Mcu_RawResetType Mcu_GetResetRawValue(void);
Mô tả

Trả về giá trị raw của reset status register (RCC_CSR trên STM32F4/F1). Dùng khi cần chi tiết hơn so với Mcu_GetResetReason — ví dụ phân biệt IWDG vs WWDG reset mà cả hai đều map vào MCU_WATCHDOG_RESET.

SID 0x07 Mcu_PerformReset
Synchronous Non Reentrant Standard
Prototype
void Mcu_PerformReset(void);
Mô tả

Thực hiện software reset MCU. Trên STM32, implement bằng cách ghi SCB->AIRCR với SYSRESETREQ bit. Function không return — MCU sẽ reset trước khi function trả về.

Constraints
  • Chỉ available khi McuPerformResetApi = TRUE.
  • Phải flush mọi pending operation (Nvm write, Dem report...) trước khi gọi — nếu không sẽ mất dữ liệu.
SID 0x08 Mcu_SetMode
Synchronous Non Reentrant Standard
Prototype
void Mcu_SetMode(Mcu_ModeType McuMode);
Mô tả

Chuyển MCU sang mode mới (RUN / SLEEP / STOP / STANDBY hoặc các mode custom). Trên STM32 thường wrap quanh __WFI(), __WFE(), hoặc set bit trong PWR_CR.

Constraints
  • Mode ID phải tồn tại trong McuModeSettingConf, nếu không sẽ trigger MCU_E_PARAM_MODE.
  • Khi vào Stop/Standby, function "trở về" sau wakeup — phải lường trước flow control.
SID 0x09 Mcu_GetVersionInfo
Synchronous Reentrant Standard
Prototype
void Mcu_GetVersionInfo(Std_VersionInfoType* versioninfo);
Mô tả

Điền vendor ID, module ID, software version vào struct. Dùng cho diagnostic và verification. Chỉ available khi McuVersionInfoApi = TRUE.

SID 0x0A Mcu_GetRamState
Synchronous Reentrant Standard
Prototype
Mcu_RamStateType Mcu_GetRamState(void);
Mô tả

Trả về trạng thái của RAM (VALID / INVALID). Dùng cho MCU có hardware support cho RAM integrity check (ECC, parity). Trên STM32F4/F1, hardware không hỗ trợ trực tiếp, nên parameter này thường set McuGetRamStateApi = FALSE và service không available.

Constraints
  • Chỉ available khi McuGetRamStateApi = TRUE.
  • Nếu HW không hỗ trợ, parameter phải FALSE — đừng implement fake.
Lưu ý về Service ID: các giá trị hex 0x00..0x0A là Service ID dùng trong DET reporting. Khi DET hook nhận callout, đối số ApiId chính là các giá trị này. Implementation thường define macro như MCU_INIT_SERVICE_ID, MCU_INITCLOCK_SERVICE_ID... trong file internal header để code dễ đọc hơn.

Error reporting

DET / DEM Errors

Development Errors (DET)

DET errors là lỗi do bug trong code (sai parameter, sai trình tự gọi, gọi khi chưa init). Chỉ được report khi McuDevErrorDetect = TRUE — production build thường tắt để tiết kiệm code size.

Error nameMô tảAPI trigger
MCU_E_PARAM_CONFIG API service gọi với ConfigPtr không hợp lệ Mcu_Init
MCU_E_PARAM_CLOCK ClockSetting vượt ngoài range cấu hình Mcu_InitClock
MCU_E_PARAM_MODE McuMode không tồn tại trong McuModeSettingConf Mcu_SetMode
MCU_E_PARAM_RAMSECTION RamSection ID vượt range Mcu_InitRamSection
MCU_E_PLL_NOT_LOCKED Mcu_DistributePllClock gọi khi PLL chưa lock Mcu_DistributePllClock
MCU_E_UNINIT API gọi trước khi Mcu_Init Mọi API trừ Mcu_Init
MCU_E_PARAM_POINTER NULL pointer pass vào API cần pointer hợp lệ Mcu_GetVersionInfo
MCU_E_INIT_FAILED Mcu_Init thất bại do invalid config (extended) Mcu_Init

Production Errors (DEM)

Theo SWS R25-11, MCU Driver có một extended production error duy nhất:

Event nameTrigger conditionReference
MCU_E_CLOCK_FAILURE Phát hiện clock source failure (HSE lost detection — CSS trigger trên STM32) SWS_Mcu_00257, SWS_Mcu_00258
SWS_Mcu R25-11: "If the clock failure is detected with other HW mechanisms e.g. the generation of a trap, this notification shall be disabled and the failure reporting shall be done outside the MCU driver." Trên STM32, CSS trigger NMI — nếu bạn xử lý NMI handler riêng để report lỗi, phải tắt McuClockSrcFailureNotification trong config để tránh double report.

Cơ chế DET callout

// Inside Mcu_InitClock implementation:
if (ClockSetting >= MCU_NUMBER_OF_CLOCK_SETTINGS) {
#if (MCU_DEV_ERROR_DETECT == STD_ON)
    (void)Det_ReportError(
        MCU_MODULE_ID,                  // 101
        MCU_INSTANCE_ID,                // 0
        MCU_INITCLOCK_SERVICE_ID,       // 0x02
        MCU_E_PARAM_CLOCK               // error code
    );
#endif
    return E_NOT_OK;
}
Pattern catch DET sớm: Trong development build, implement Det_ReportError breakpoint hoặc log để bắt mọi DET callout. Một DET error không bao giờ là "warning" — nó là bug thực sự trong integration code. Fix ngay khi thấy, đừng đợi production fail.

Use cases

Use case tích hợp thực tế

CASE / 01
Standard startup sequence — Body Control Module (BCM)
Mcu_Init Mcu_InitClock Mcu_GetPllStatus Mcu_DistributePllClock STM32F4 @ 168 MHz

Kịch bản

ECU điều khiển khoang xe (BCM): điều khiển đèn, kính, khóa cửa. Yêu cầu CPU 168 MHz (F4) để chạy CAN, LIN, và stack diagnostic UDS. Startup phải hoàn tất dưới 100ms.

Call sequence trong EcuM

// Step 1: MCU init (vào sớm nhất trong InitZero)
Mcu_Init(&Mcu_Config);   // hoặc NULL_PTR nếu PC variant

// Step 2: Init clock setting "168MHz_HSE"
if (Mcu_InitClock(0) != E_OK) {
    // fall back hoặc trigger fail-safe
}

// Step 3: Poll PLL lock với timeout
uint32 timeout = MCU_PLL_LOCK_TIMEOUT;
while ((Mcu_GetPllStatus() != MCU_PLL_LOCKED) && (timeout--));

if (Mcu_GetPllStatus() != MCU_PLL_LOCKED) {
    // PLL lock failed → safe-state
}

// Step 4: Distribute PLL clock to bus
(void)Mcu_DistributePllClock();

// Step 5: Read reset reason for diagnostics
Mcu_ResetType reason = Mcu_GetResetReason();
if (reason == MCU_WATCHDOG_RESET) {
    // log to Dem, possibly enter safe-mode
    Dem_SetEventStatus(DemConf_DemEventParameter_WDG_RESET, DEM_EVENT_STATUS_FAILED);
}

// Step 6: Now EcuM continues with Port_Init, Dio_Init, etc.

Bẫy thường gặp

  • Quên Mcu_DistributePllClock → SYSCLK vẫn dùng HSI 16 MHz dù PLL đã lock. UART/CAN baudrate sai.
  • Timeout PLL lock quá ngắn → false-fail trên board lạnh (HSE startup chậm hơn ở -40°C).
CASE / 02
Reset reason logging — Diagnostic ECU
Mcu_GetResetReason Mcu_GetResetRawValue Dem integration

Kịch bản

ECU diagnostic gateway cần ghi nhận mọi reset bất thường để báo cáo qua DTC khi thợ scan xe. Phân biệt được POR (bình thường) vs Watchdog reset (bug) vs External reset (có thể do EMI).

Configuration outline

  • McuResetSettingConf ánh xạ từng reset reason → DEM event ID.
  • Bật McuPerformResetApi = TRUE để có thể trigger reset từ diagnostic command.

Call pattern sau wakeup

void DiagECU_PostReset(void) {
    Mcu_ResetType  reason = Mcu_GetResetReason();
    Mcu_RawResetType  raw = Mcu_GetResetRawValue();

    switch (reason) {
        case MCU_POWER_ON_RESET:
            // normal — just log info
            break;
        case MCU_WATCHDOG_RESET:
            // raw value tells IWDG vs WWDG on STM32F4
            if (raw & RCC_CSR_IWDGRSTF) {
                Dem_SetEventStatus(DEM_EVT_IWDG_RST, DEM_EVENT_STATUS_FAILED);
            } else if (raw & RCC_CSR_WWDGRSTF) {
                Dem_SetEventStatus(DEM_EVT_WWDG_RST, DEM_EVENT_STATUS_FAILED);
            }
            break;
        case MCU_SW_RESET:
            // intentional, e.g. after firmware update
            break;
        case MCU_RESET_UNDEFINED:
        default:
            Dem_SetEventStatus(DEM_EVT_UNKNOWN_RST, DEM_EVENT_STATUS_FAILED);
    }
}

Lưu ý

Reset flag trong RCC_CSR phải được clear sau khi đọc — nếu không, mọi lần boot sẽ thấy flag cũ. Nhiều implementation MCU Driver tự động clear trong Mcu_GetResetReason, nhưng cần verify với vendor doc — không phải lúc nào cũng vậy.

CASE / 03
Low-power wakeup — Smart Junction Box
Mcu_SetMode BswM integration EcuM_Sleep STM32F1 @ 72 MHz

Kịch bản

ECU phụ trong xe chạy pin: khi xe khóa, ECU phải vào Stop mode chỉ tiêu thụ µA. Khi có CAN wake-up frame, MCU phải wake và xử lý trong vài ms.

Configuration outline

  • Định nghĩa 2 mode trong McuModeSettingConf: ID 0 = RUN, ID 1 = STOP.
  • BswM rule chuyển sang STOP khi nhận tín hiệu sleep từ ComM/Nm.

Call pattern

// Triggered by BswM when CommunicationMode = NO_COMMUNICATION
void BswM_Action_EnterSleep(void) {
    // 1. Disable peripherals not needed in sleep
    CanIf_SetControllerMode(0, CAN_CS_SLEEP);

    // 2. Configure wakeup sources (CAN RX pin EXTI...)
    EcuM_SelectShutdownTarget(ECUM_STATE_SLEEP, 0);

    // 3. Switch MCU mode → STOP
    Mcu_SetMode(MCU_MODE_STOP);
    // MCU enters WFI / STOP — code execution paused here
    // ...
    // On wakeup, execution resumes RIGHT HERE

    // 4. Re-init clock (STOP mode disabled PLL)
    Mcu_InitClock(0);
    while (Mcu_GetPllStatus() != MCU_PLL_LOCKED);
    (void)Mcu_DistributePllClock();

    // 5. Notify EcuM about wakeup
    EcuM_CheckWakeup(WAKEUP_SRC_CAN);
}

Bẫy thường gặp

  • Trên STM32F1: Stop mode reset HSE/PLL, sau wakeup MCU chạy HSI 8 MHz mặc định. Nếu không re-init clock, CAN/UART sẽ sai baudrate.
  • Trên STM32F4: Tương tự nhưng có thêm tùy chọn STOP với regulator low-power. Phải verify wake-up time đáp ứng yêu cầu hệ thống (50-100µs vs vài ms).
CASE / 04
Multiple clock setting cho safety — Engine Control Module
2× McuClockSettingConfig CSS enabled ASIL-B target

Kịch bản

EMS yêu cầu degraded mode khi HSE fail (ví dụ thạch anh chết do EMI hoặc vibration). Bình thường chạy 168 MHz từ HSE, fallback xuống 16 MHz HSI khi mất HSE — vẫn đủ điều khiển spark cơ bản để xe về xưởng.

Configuration outline

  • McuClockSettingId = 0: "Normal_168MHz_HSE_PLL".
  • McuClockSettingId = 1: "Safe_16MHz_HSI_NoPll".
  • McuClockSrcFailureNotification = TRUE để CSS trigger DEM event.

Switch flow

// NMI Handler triggered by HSE Clock Security System
void Mcu_ClockFailureNotification(void) {
    // Implemented as DEM event callback
    Dem_SetEventStatus(DEM_EVT_CLOCK_FAILURE, DEM_EVENT_STATUS_FAILED);

    // Switch to safe clock setting
    if (Mcu_InitClock(1) == E_OK) {  // Safe mode
        // HSI doesn't need PLL — skip distribute step
        // Notify upper layer to enter limp-home mode
        BswM_RequestMode(BSWM_MODE_LIMP_HOME);
    } else {
        // Catastrophic failure → reset
        Mcu_PerformReset();
    }
}

Lưu ý

ASIL-rated MCU Driver từ vendor (NXP, Vector, Elektrobit) thường có DDV (Driver Development Verification) report — cần dùng version đã qualified, không tự implement. Self-developed MCAL trên STM32 chỉ phù hợp với QM (Quality Management), không đạt ASIL-B trở lên trừ khi có safety case đầy đủ.

CASE / 05
Dynamic frequency scaling — Battery Management System
3× McuClockSettingConfig Runtime clock switch STM32F4

Kịch bản

BMS đo dòng và điện áp 256 cell pin. Khi xe ở chế độ active (sạc/xả), cần CPU full speed để chạy thuật toán SOC/SOH. Khi xe nghỉ, giảm CPU xuống để kéo dài pin 12V phụ.

Configuration outline

  • Clock setting 0: 168 MHz full performance.
  • Clock setting 1: 84 MHz balanced.
  • Clock setting 2: 16 MHz idle.

Switch trigger

// Application requests low power → BswM action
void BswM_Action_LowPowerClock(void) {
    // 1. Stop time-critical peripherals
    CanIf_SetControllerMode(0, CAN_CS_STOPPED);

    // 2. Switch clock
    Mcu_InitClock(2);  // 16 MHz idle
    // Note: HSI doesn't need PllDistribute on STM32

    // 3. Restart peripherals with new clock
    CanIf_SetControllerMode(0, CAN_CS_STARTED);

    // 4. UART baudrate auto-recalculated by upper-layer
}

Bẫy thường gặp

  • Quên reconfig peripheral sau switch clock → UART/SPI/CAN baudrate sai. Stop trước, restart sau.
  • Flash wait state phải đổi trước khi tăng clock và sau khi giảm clock. Implementation MCU Driver phải xử lý đúng — verify với vendor!
CASE / 06
Software reset từ diagnostic — UDS service 0x11
Mcu_PerformReset Dcm integration NvM flush

Kịch bản

Sau khi flash firmware mới qua UDS, tool thợ máy gửi service 0x11 (ECUReset). MCU phải reset để chạy firmware mới — nhưng trước đó phải flush NvM và đảm bảo không có ghi đang dở.

Implementation

Std_ReturnType Dcm_EcuReset_Indication(void) {
    // 1. Send positive response BEFORE reset
    Dcm_SendPositiveResponse();

    // 2. Wait for transmission complete (max ~100ms)
    PduR_WaitForTransmissionDone();

    // 3. Flush NvM — critical to not lose calibration data
    NvM_WriteAll();
    while (NvM_GetStatus() != NVM_REQ_OK);

    // 4. Disable interrupts and trigger reset
    SchM_Enter_Mcu_MCU_EXCLUSIVE_AREA_0();
    Mcu_PerformReset();
    // Never returns

    return E_OK;  // unreachable
}

Bẫy thường gặp

  • Gọi Mcu_PerformReset trước khi UDS response gửi ra → tool báo timeout.
  • Không flush NvM → mất config khi reset. Đảm bảo NvM_WriteAll hoàn tất trước.
Pattern chung khi tích hợp MCU Driver: Luôn nghĩ về MCU Driver như "abstraction của RCC/PWR/SCB" — code application không bao giờ touch register trực tiếp. Mọi clock change, reset, mode change đều route qua API. Khi review code, search keyword "RCC->" hoặc "PWR->" — mọi xuất hiện ngoài Mcu.c là vi phạm AUTOSAR layering.

Init dependency

Init order & Dependency

MCU Driver là root dependency

Trong AUTOSAR BSW stack, MCU Driver là một trong những module init đầu tiên. Hầu hết các MCAL khác (Port, Dio, Spi, Can, Adc...) cần clock đã ổn định, mà clock được set up bởi MCU Driver.

EcuM Startup Sequence (simplified) ══════════════════════════════════════════ [1] Start-up code (vector table, stack, .data, .bss) ↓ [2] Mcu_Init() — ROOT — must be first MCAL ↓ [3] Mcu_InitClock(0) — configure PLL ↓ [4] Wait Mcu_GetPllStatus() == LOCKED ↓ [5] Mcu_DistributePllClock() ↓ [6] Port_Init() — GPIO config (depends on MCU clock) ↓ [7] Dio_Init(), Gpt_Init(), Adc_Init(), ... ↓ [8] Spi_Init(), Can_Init(), ... ↓ [9] ECU Abstraction Layer init ↓ [10] Service Layer: Os, Com, Dcm, ... ↓ [11] RTE init → SWC start

Module phụ thuộc MCU Driver

  • Tất cả MCAL khác — vì cần clock được set up trước.
  • EcuM — gọi Mcu_GetResetReason để xác định startup path.
  • BswM — gọi Mcu_SetMode khi chuyển ECU mode.
  • Dcm (Diagnostic Communication Manager) — gọi Mcu_PerformReset cho service 0x11.

Module MCU Driver phụ thuộc

  • DET — để report development error (nếu McuDevErrorDetect = TRUE).
  • DEM — để report MCU_E_CLOCK_FAILURE (nếu McuClockSrcFailureNotification = TRUE).
  • SchM — để protect critical section (read-modify-write trên internal state).
  • Không phụ thuộc module nào trong cùng MCAL layer.
Bẫy init order kinh điển: Quên gọi Mcu_DistributePllClock sau khi PLL lock → các MCAL khác init thành công về mặt API nhưng clock thực tế vẫn là HSI/HSE thấp. UART/CAN baudrate calculation đúng theo config (giả định CPU 168 MHz) nhưng register set lại dựa trên thực tế (16 MHz) → mọi truyền thông sai. Bug rất khó debug vì code không lỗi rõ ràng.

Memory & Performance

Memory mapping sections

MCU Driver tuân thủ AUTOSAR memory mapping (file MemMap.h). Các section điển hình:

SectionNội dung
MCU_START_SEC_CODE / STOP_SEC_CODECode thực thi của các hàm Mcu_*
MCU_START_SEC_CONST_*BITConfig struct hard-coded (PC variant)
MCU_START_SEC_VAR_INIT_UNSPECIFIEDInternal state khởi tạo (init flag, current mode)
MCU_START_SEC_VAR_NOINIT_UNSPECIFIEDInternal state không cần init (cache values)
MCU_START_SEC_VAR_FAST_INIT_UNSPECIFIEDState truy cập tần suất cao (nếu vendor support)

Footprint xấp xỉ trên STM32F4/F1

Tùy implementation (vendor MCAL hoặc self-developed), số liệu có thể dao động:

  • ROM: 2-5 KB code + 200-500 B const config (PC variant, all APIs enabled).
  • RAM: 20-100 B internal state.
  • Tắt API không dùng (McuPerformResetApi = FALSE, McuGetRamStateApi = FALSE) giảm được ~10-20% ROM.
  • DET disable (McuDevErrorDetect = FALSE) giảm thêm ~15-25% ROM trong production build.

Timing constraints

  • Mcu_Init: thường < 100 µs (chỉ set vài register).
  • Mcu_InitClock: vài ms trên STM32F4 (PLL lock ~200 µs + flash latency setup), tương tự F1.
  • Mcu_GetPllStatus: vài chu kỳ CPU (đọc 1 register).
  • Mcu_PerformReset: không return — MCU reset trong vài µs.
  • Mcu_SetMode (vào STOP/STANDBY): tức thì, nhưng wakeup latency 50-100 µs (F4) / 5-10 µs (F1).
Lưu ý timing cho safety-critical: Nếu ECU có deadline cứng (ví dụ EMS phải phản hồi crank signal dưới 1ms), tính cả thời gian wake-up từ STOP. Trên STM32F1, wake-up nhanh hơn (chỉ vài µs) vì kiến trúc đơn giản. STM32F4 chậm hơn do flash trickier latency.

Integration

Integration với upper layer

EcuM integration

EcuM gọi MCU Driver trong các state chính:

  • StartPreOS: Mcu_Init, Mcu_InitClock, Mcu_GetResetReason.
  • Sleep transition: Mcu_SetMode (sang sleep mode tương ứng).
  • Wakeup: re-init clock sau khi rời low-power.
  • GoOffOne: Mcu_PerformReset trước khi shutdown ECU.

BswM integration

BswM có thể trigger Mcu_SetMode thông qua action list khi nhận mode request từ application. Ánh xạ thường configure qua tool, không cần code C trực tiếp.

Dcm integration

Dcm gọi Mcu_PerformReset để xử lý UDS service 0x11 (ECUReset). Cần ensure transmission của positive response hoàn tất trước reset.

Quy ước naming cần nhớ

  • Public API: Mcu_<verb><Object>() — Mcu_Init, Mcu_InitClock, Mcu_SetMode...
  • Configuration container: Mcu<Object>Conf — McuClockSettingConfig, McuModeSettingConf...
  • Type: Mcu_<Object>Type — Mcu_ConfigType, Mcu_ClockType...
  • Error: MCU_E_<Reason> — MCU_E_PARAM_CLOCK, MCU_E_UNINIT...
Compiler abstraction: Code MCU Driver chuẩn AUTOSAR phải dùng macros từ Compiler.h (như FUNC(), P2CONST(), VAR()) thay vì C keyword trực tiếp. Điều này cho phép port giữa các compiler khác nhau (Keil, GCC, IAR, Greenhills).

F4 vs F1

Khác biệt STM32F4 vs STM32F1

Mặc dù cùng dùng MCU Driver API chuẩn, implementation cho STM32F4 và STM32F1 khác biệt đáng kể ở tầng hardware. Bảng dưới tóm tắt các điểm cần lưu ý khi config tool hoặc viết MCAL self-developed.

So sánh hardware

Tính năngSTM32F1xxSTM32F4xx
CPU core Cortex-M3 Cortex-M4F (FPU)
Max system clock 72 MHz 100-180 MHz (model-dependent)
PLL structure PREDIV + MUL (đơn giản) M / N / P / Q (4 chia/nhân)
HSE crystal range 4-16 MHz 4-26 MHz
HSI frequency 8 MHz 16 MHz
Second PLL (audio) Không có PLLI2S (chỉ một số model F4)
USB 48 MHz source PLL /1.5 (cố định) PLL Q (linh hoạt)
Voltage scaling Không có VOS Scale 1/2/3 (tiết kiệm điện)
Flash latency max 2 WS 7 WS (180 MHz, VOS=1)
Reset reason bits (RCC_CSR) 7 nguyên nhân 8 nguyên nhân (thêm BOR)
Stop mode wakeup latency ~5 µs ~13-100 µs (regulator dependent)
Standby current ~2-3 µA ~2.5 µA (with RTC)

Ảnh hưởng tới config

Khi config MCU Driver qua tool (tresos / DaVinci), một số khác biệt phải khai báo đúng dòng MCU:

STM32F1 — PLL config đơn giản

// STM32F1: SYSCLK = HSE_or_HSI / PREDIV * MUL
// Example: HSE 8 MHz → 72 MHz
HSE_freq      = 8 MHz
PLL_source    = HSE
PLL_mul       = 9      // → 8 MHz × 9 = 72 MHz
AHB_prescaler = 1
APB1_prescal  = 2      // max 36 MHz
APB2_prescal  = 1      // max 72 MHz
Flash_latency = 2      // 2 WS for >48 MHz

STM32F4 — PLL config phức tạp hơn

// STM32F4: SYSCLK = (HSE / M * N) / P
// Example: HSE 8 MHz → 168 MHz (F407) or 100 MHz (F411)
HSE_freq      = 8 MHz
PLL_source    = HSE
PLL_M         = 8      // 8/8 = 1 MHz (must be 1-2 MHz)
PLL_N         = 336    // 1 * 336 = 336 MHz VCO
PLL_P         = 2      // 336/2 = 168 MHz SYSCLK
PLL_Q         = 7      // 336/7 = 48 MHz USB
AHB_prescaler = 1
APB1_prescal  = 4      // max 42 MHz on F407
APB2_prescal  = 2      // max 84 MHz on F407
Flash_latency = 5      // 5 WS @ 168 MHz, VOS=1
VOS_scale     = 1      // for max performance

Mcu_GetResetRawValue — bit layout khác nhau

Cả hai dòng đều có thanh ghi RCC_CSR ghi reset cause, nhưng bit position khác nhau. Implementation phải dùng correct mask cho từng MCU:

Window watchdogbit 30 (WWDGRSTF)bit 30 (WWDGRSTF)
Independent watchdogbit 29 (IWDGRSTF)bit 29 (IWDGRSTF)

Mapping clock setting ID khi switch dòng: Nếu codebase phải chạy được trên cả F1 và F4 (rare nhưng có), giữ cùng McuClockSettingId cho các use case tương đương: ID 0 = "max performance", ID 1 = "balanced", ID 2 = "low power". Application code chỉ gọi Mcu_InitClock(0..2), không cần biết MCU nào.

Debug

Checklist debug & tích hợp

Khi MCU không boot hoặc clock sai

  1. Verify file generated — mở Mcu_Cfg.h và Mcu_Cfg.c, kiểm tra giá trị PLL M/N/P/Q khớp với clock setting bạn config trong tresos. Một con số sai làm sai cả PLL.
  2. Check init order trong EcuM — đảm bảo Mcu_Init gọi trước Mcu_InitClock, và DistributePllClock gọi sau khi PLL lock. Nếu dùng EcuM tool-generated, mở EcuM_Callout_Stubs.c kiểm tra.
  3. Bật McuDevErrorDetect — set hook để bắt mọi DET error. Place breakpoint trong Det_ReportError để xem service nào gọi với error code gì.
  4. Đo clock bằng MCO — output PLLCLK ra chân MCO (PA8 trên F4, PA8 trên F1) với prescaler hợp lý, đo bằng oscilloscope. Đây là cách không thể nói dối để biết clock thực tế.
  5. Verify HSE ready — kiểm tra RCC_CR.HSERDY bằng debugger. Nếu mãi không lên, có thể: thạch anh chết, cap matching sai, layout PCB tệ.
  6. Check Flash latency — đọc FLASH_ACR.LATENCY. Nếu sai (quá thấp khi clock cao), MCU sẽ hard-fault hoặc treo.
  7. Read Mcu_GetResetReason ở đầu main — biết MCU đến state hiện tại từ đâu. Watchdog reset thường chỉ ra clock chậm hơn kỳ vọng.

Khi UART/CAN/SPI baudrate sai

  1. Verify SystemCoreClock (nếu dùng CMSIS variable) — bị sai sau khi switch clock mà không gọi update.
  2. Check Mcu_DistributePllClock đã gọi chưa — đây là bug #1 cho baudrate sai mà PLL vẫn lock.
  3. Verify APB prescaler — nhớ rằng timer trên APB có quy tắc đặc biệt (×2 khi prescaler > 1).
  4. Stop / start peripheral khi switch clock — nếu dynamic frequency scaling, peripheral phải reconfig sau khi clock đổi.

Khi DET error trigger liên tục

  1. Identify ApiId + ErrorId trong Det_ReportError parameter — biết exactly service nào và lý do gì.
  2. MCU_E_UNINIT → ai đó gọi API trước Mcu_Init. Check init order.
  3. MCU_E_PARAM_CLOCK → ClockSetting argument lớn hơn số config. Check argument value vs McuClockSettings count.
  4. MCU_E_PLL_NOT_LOCKED → DistributePllClock gọi trước khi lock. Check timeout loop có exit đúng không.
Quy tắc vàng debug MCU Driver: 90% bug đến từ một trong ba nguyên nhân — sai init order, sai config value trong tresos, hoặc quên DistributePllClock. Verify ba điều này trước khi nghi ngờ implementation MCAL hay hardware.
